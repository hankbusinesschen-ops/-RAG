# 金控年報 RAG 智慧問答系統 — 技術說明報告

> 適用對象：非純技術背景主管  
> 報告日期：2026-04-07  
> 撰寫人：Hank Chen

---

## 一、這個系統在做什麼？

**用一句話說**：把金融機構的年度報告（PDF）「餵給」AI，讓使用者可以用自然語言提問，系統從年報中精準找出答案並附上出處，同時確保答案不會「亂講」。

**解決的核心問題**：年報動輒 200–300 頁，人工翻查費時，且金控旗下有多家子公司（富邦人壽、台北富邦銀行、富邦證券等），資料容易混淆。系統原始版本準確率為 **0%**，主因是把子公司財報與金控合併財報混在一起。

                    ┌──────▼──────┐

## 三、核心技術名詞詳解

### 3.1 RAG（Retrieval-Augmented Generation，檢索增強生成）

**是什麼**  
RAG 是一種 AI 問答架構，將「查資料」和「寫答案」分成兩個步驟：
1. **檢索**（Retrieval）：先從文件庫找出最相關的段落
2. **生成**（Generation）：把找到的段落交給 AI，讓它根據這些段落寫出答案

**為什麼用 RAG，不直接問 AI？**

| 問題 | 直接問 AI | RAG |
|------|-----------|-----|
| AI 會不會亂講？ | 會（幻覺問題）| 有根據，可驗證來源 |
| 能否用最新年報？ | 不行（AI 訓練資料有截止日期）| 可以，即時從文件檢索 |
| 子公司資料會混淆嗎？ | 會 | 不會（有實體標記區分） |
| 答案可追溯嗎？ | 不可 | 可（顯示出自第幾頁） |

---

### 3.2 Gemini（Google 的大型語言模型）

**是什麼**  
Google 開發的 AI 模型系列，本系統使用三種 Gemini 功能：

| 功能 | 模型 | 用途 |
|------|------|------|
| **LLM 問答生成** | `gemini-2.0-flash` | 讀取段落後撰寫中文答案 |
| **文字向量化** | `gemini-embedding-2-preview` | 把文字轉成數字向量，用於相似度搜尋 |
| **圖片理解（Vision）** | Gemini Vision API | 處理旋轉或模糊的 PDF 頁面，OCR 提取文字 |

**為什麼選 Gemini？**
- 原生支援中文，對金融術語理解佳
- Vision API 可直接處理圖片頁面，不需額外 OCR 工具
- Embedding 模型支援 **8192 tokens**（同類產品中最大），適合長段落
- 同一個 API Key 涵蓋所有功能，成本與整合複雜度低

---

### 3.3 Embedding（向量化）

**是什麼**  
將文字轉換成一串數字（向量），讓電腦可以計算兩段文字的「語意相似度」。

例如：
- 「稅後淨利」和「淨利潤」在向量空間中距離很近 → 視為同一意思
- 「富邦人壽保費收入」和「台北富邦銀行存款」距離較遠 → 不同主題

**非對稱 Embedding 策略（本系統特有）**

| 情境 | Task Type | 原因 |
|------|-----------|------|
| 建索引時（文件段落） | `retrieval_document` | 讓文件向量最大化表達文件內容 |
| 使用者提問時 | `question_answering` | 讓問題向量更接近「問問題」的語意模式 |

這個設計讓「問題」和「文件」各自在最適合的向量空間，檢索準確度更高。

---

### 3.4 FAISS（Facebook AI Similarity Search，向量資料庫）

**是什麼**  
Meta（Facebook）開源的向量搜尋函式庫。把所有段落的向量存起來，提問時快速找出最相近的向量（即最相關的段落）。

**為什麼選 FAISS？**
- **本地執行**：不需要架設外部伺服器（Pinecone、Weaviate 需要），適合初期開發與資安考量
- **速度快**：L2 距離計算，毫秒級查詢
- **支援合併**：多份年報的索引可以直接合併，無需重新嵌入
- **快取機制**：同一份 PDF 只需建一次索引，節省 API 費用

---

### 3.5 BM25（Best Match 25，關鍵字搜尋）

**是什麼**  
一種經典的關鍵字排序演算法，根據詞頻與文件長度計算相關性。Google 早期搜尋引擎的核心邏輯。

**為什麼還需要 BM25？（搭配向量搜尋）**

向量搜尋擅長「語意理解」，BM25 擅長「精確關鍵字匹配」：

| 查詢範例 | FAISS 向量 | BM25 |
|---------|-----------|------|
| 「EPS 是多少」 | 語意理解，找到利潤相關段落 | 精確匹配「EPS」字串 |
| 「113年度富邦人壽初年度保費」 | 可能語意漂移 | 精確匹配每個關鍵詞 |

兩者互補，一起用效果比單獨用任一種好 30-50%。

---

### 3.6 RRF（Reciprocal Rank Fusion，倒數排名融合）

**是什麼**  
把多個搜尋通道的結果融合成一個排名的演算法。公式：

```
最終分數 = 1/(60 + 在通道A的排名) + 1/(60 + 在通道B的排名) + ...
```

**本系統三個搜尋通道**：

| 通道 | 技術 | 角色 |
|------|------|------|
| 通道 1 | FAISS 語意向量 | 理解問題意圖 |
| 通道 2 | BM25 關鍵字 | 精確詞彙匹配 |
| 通道 3 | 圖片語意向量 | 找表格圖表所在頁面 |

**為什麼選 RRF 而不是簡單加權平均？**  
RRF 只看「排名」不看「分數絕對值」，避免三個通道分數量級不同（FAISS 輸出 L2 距離、BM25 輸出 TF-IDF 分數）造成的偏差。是學術界驗證過的最佳融合方法之一。

---

### 3.7 TOC 目錄解析（Table of Contents Parser）

**是什麼**  
自動讀取 PDF 年報的目錄頁，建立「章節號 → 頁碼」的對應表，讓每個段落都知道自己屬於哪個章節。

**為什麼重要？**

沒有目錄解析的問題：
- 「富邦人壽 2023 年初年度保費」和「富邦金控合併報表」的段落混在一起，分不清楚

有目錄解析之後：
- 每個段落帶有標籤：`section_path = "4 營運概況 > 4.2 富邦人壽 > 4.2.1 保費收入"`
- 檢索時可以根據問題類型，優先搜尋對應章節

---

### 3.8 Chunking（文件分塊）

**是什麼**  
把一份 300 頁的年報切成很多小段落（每段約 1000 字），讓 AI 可以精準定位答案所在位置。

**關鍵設計參數**：

| 參數 | 值 | 說明 |
|------|-----|------|
| `chunk_size` | 1000 字 | 每個段落最大長度 |
| `chunk_overlap` | 200 字 | 相鄰段落重疊，避免答案被切斷 |
| 分割優先順序 | 段落 → 句子 → 逗號 | 盡量在自然斷點切割 |
| 表格處理 | 整體保留，不切割 | 表格切一半會失去數據意義 |

---

### 3.9 實體感知 Metadata（Entity-Aware Metadata）

**是什麼**  
給每個段落貼上「來自哪家公司、哪個層級」的標籤。

```
entity_level = "consolidated"   ← 金控合併報表
entity_level = "subsidiary"     ← 子公司報表
entity = "富邦人壽"              ← 具體子公司名稱
```

**為什麼這是最核心的改進？**  
原系統 0% 準確率的根本原因：問「富邦金控合併稅後淨利」，系統卻拿「富邦人壽子公司淨利」來回答，數字相差數倍。

加上實體標記後，系統能根據問題關鍵字自動提升正確層級的段落排名：
- 問「金控整體」→ 優先 `consolidated` 資料
- 問「富邦人壽」→ 優先 `entity = 富邦人壽` 資料

---

### 3.10 LLM Reranker（大型語言模型重排序）

**是什麼**  
從 RRF 融合後的候選段落（前 20 個）中，再用 AI 重新判斷「這些段落真的跟問題相關嗎？」，取出最相關的前 5 個送給 LLM 生成答案。

**為什麼需要 Reranker？**

向量搜尋和 BM25 都是統計方法，無法真正「理解」段落。例如：

- 查詢：「富邦 2023 年 EPS？」
- BM25 可能找到：提到「EPS」的一般說明段落（不含數字）
- Reranker 會識別：「這段沒有 2023 年的實際數值，排名下調」

**成本考量**：Reranker 只對前 20 個候選做判斷，API 費用極低。

---

### 3.11 幻覺檢測（Hallucination Guard）

**是什麼**  
「幻覺」（Hallucination）是指 AI 自己編造沒有根據的答案。本系統有兩層防護：

**第一層：檢索相關度檢查**
- 如果找到的最相關段落和問題的向量距離 > 1.2（閾值），代表年報中根本沒有這個資訊
- 系統直接回覆：「根據所提供的年報內容，無法找到此資訊。」

**第二層：答案錨定驗證**
- 生成答案後，再用 AI 逐句核對：「這句話的根據在哪個段落？」
- 如果找不到對應依據，標記為幻覺風險

**風險等級顯示**：

| 等級 | 條件 | UI 顯示 |
|------|------|---------|
| 低風險 | 兩層都通過 | 綠色提示 |
| 中風險 | 其中一層有疑慮 | 黃色警告 |
| 高風險 | 兩層都有問題 | 紅色警告 |

---

### 3.12 PyMuPDF + pdfplumber（PDF 解析工具）

**是什麼**

| 工具 | 角色 | 特長 |
|------|------|------|
| **PyMuPDF（fitz）** | 主要文字提取 | 速度快，支援旋轉偵測，可渲染頁面圖片 |
| **pdfplumber** | 表格提取 | 精確識別表格格線，輸出為 Markdown 表格 |
| **Gemini Vision** | 備援 OCR | 當文字品質差或頁面旋轉時，用 AI 直接看圖片提取文字 |

**觸發 Vision 備援的條件**：
- 頁面旋轉角度 > 45 度
- 提取的中文字元數 < 50 個（代表可能是掃描圖片）

---

### 3.13 jieba（中文分詞）

**是什麼**  
中文沒有空格分隔詞彙，jieba 是最廣泛使用的開源中文斷詞函式庫，把「富邦人壽保費收入」切成「富邦人壽 / 保費 / 收入」。

**為什麼需要分詞？**  
BM25 關鍵字搜尋需要先把文字切成獨立詞彙才能計算詞頻。

**客製化金融詞典**：  
系統預先將金融術語（ROE、資本適足率、初年度保費等）加入 jieba 詞典，避免被錯誤切割（例如「資本適足率」不應被切成「資本 / 適 / 足 / 率」）。

---

### 3.14 LangChain / LCEL（LangChain Expression Language）

**是什麼**  
LangChain 是一個 AI 應用開發框架，LCEL 是其中的「管線串接語法」，讓各個 AI 元件像積木一樣組合。

```python
# 範例：問答 Chain 只需一行
chain = prompt_template | gemini_llm | output_parser
```

**為什麼用 LangChain？**
- 模組化設計：Prompt、LLM、Retriever 各自獨立，方便替換（例如未來改成 Claude 或 GPT）
- 完整生態系：FAISS、BM25 都有 LangChain 整合
- 開發效率高：減少重複樣板程式碼

---

### 3.15 Streamlit（前端使用者介面）

**是什麼**  
Python 原生的快速 Web UI 框架，不需要前端工程師即可建立互動式介面。

**系統 UI 功能**：

| 功能 | 說明 |
|------|------|
| PDF 上傳 | 支援多份年報同時上傳 |
| 聊天介面 | 連續問答，保留對話歷史 |
| 來源展開 | 每個答案附上引用段落和頁碼 |
| 參數調整 | chunk_size、retrieve_k 等可即時調整（方便調優） |
| 批次評估頁 | 自動跑測試題集，計算準確率 |
| 風險指示燈 | 幻覺風險等級視覺化 |

**為什麼選 Streamlit？**  
快速原型驗證，無需 DevOps 部署，適合 PoC（概念驗證）階段。

## 四、技術選型決策總表

| 選擇的技術 | 評估過的替代方案 | 選擇原因 |
|-----------|---------------|---------|
| **Gemini** LLM | OpenAI GPT-4、Claude | 原生支援圖片（Vision），同一 API 整合 LLM + Embedding + Vision，初期成本較低 |
| **FAISS** 向量庫 | Pinecone、Weaviate、Chroma | 本地無伺服器，資安合規（資料不出公司），零維運成本 |
| **BM25 + RRF** 混合檢索 | 純向量搜尋 | 混合搜尋在 NLP 領域普遍優於單一方法，RRF 是標準融合算法 |
| **LLM Reranker** | CrossEncoder 模型（需下載） | 零額外部署，Gemini API 直接使用，容錯能力強 |
| **PyMuPDF + pdfplumber** | pdfminer、pypdf2、Adobe PDF API | PyMuPDF 速度最快，pdfplumber 表格最精確，兩者互補 |
| **jieba** 分詞 | CKIP、HanNLP | 最成熟開源中文分詞，自訂詞典最方便 |
| **Streamlit** | Flask + React、FastAPI | 純 Python，快速開發，適合 PoC 不需前端工程師 |
| **LangChain** | 自行實作 | 模組化可替換、生態系完整、減少重複程式碼 |

## 五、常見主管問題 Q&A

**Q1：這跟直接把年報 PDF 丟給 ChatGPT 有什麼不同？**

> ChatGPT 有 token 限制（約 128K），一份 300 頁年報約 150 萬字，根本放不下。RAG 的作法是「先找再答」，每次只送最相關的 5 個段落給 AI，不受長度限制。此外，本系統加了幻覺檢測和子公司/合併報表的區分標記，這些是通用 ChatGPT 做不到的。

**Q2：AI 的答案準確嗎？怎麼確保不會亂講？**

> 三層保護：① 幻覺檢測會比對答案是否有年報依據 ② 答案附上來源頁碼，可人工驗證 ③ 系統找不到資料時會明確說「無法找到此資訊」，而不是編造。系統有自動評估模組，用預先準備的標準題集測試，目前正在調優到穩定準確率。

**Q3：資料安全嗎？年報會上傳到哪裡？**

> 年報 PDF 處理後的向量索引儲存在**本地機器**（FAISS），不上傳到任何外部資料庫。唯一對外的是 Gemini API 調用（文字片段），若有嚴格資安需求，可評估改用本地開源模型（如 LLaMA）替換 Gemini。

**Q4：可以支援其他金控的年報嗎？**

> 可以。金控公司名稱和子公司清單放在 `config/entities.json` 設定檔，新增一家金控只需更新設定檔，不需改動程式碼。目前已預設支援富邦、國泰、中信、玉山等主要金控。

**Q5：未來可以轉為正式產品嗎？**

> 可以，擴展路徑清晰：
> 1. 前端換 FastAPI + React（Streamlit 是 PoC 用）
> 2. 向量庫換 Pinecone 或 Milvus（支援多人並發）
> 3. LLM 可替換（LangChain 模組化設計讓替換成本低）
> 4. 加入用戶認證和 API 層，即可作為 SaaS 服務

**Q6：效能如何？每次問答要等多久？**

> 主要延遲來自 Gemini API（通常 2-5 秒）。本地 FAISS 搜尋和 BM25 搜尋合計 < 0.1 秒。有多項快取機制：PDF 已建索引則直接載入（無需重新嵌入），Vision API 結果也會快取。

**Q7：為什麼選 Gemini 而不是 OpenAI GPT？**

> 主因有三：① Gemini 同一 API 涵蓋 LLM + Embedding + Vision，OpenAI 的圖片理解需額外費用 ② `gemini-embedding-2-preview` 支援 8192 tokens，比 OpenAI 的 8191 token 上限稍有優勢，且原生多模態（文字和圖片在同一向量空間）③ 初期開發成本考量。系統的 LangChain 架構支援替換 LLM，未來若有需要可以切換。
